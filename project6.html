WorkoutPal

Developing A UX-Focused Android Workout App

Inspiration and Problem Space

WorkoutPal emerged from observing a gap in fitness tracking solutions during my IAT359 Mobile Computingcourse. While existing apps offered workout logging, few combined comprehensive data tracking with real-time sensor feedback.

My vision was to create an Android application specifically tailored for weight training enthusiasts who needed more than basic rep counting.

The core challenge was balancing three competing priorities:

  1. Creating an intuitive interface for quick gym-floor logging
  2. Implementing meaningful sensor integration without clutter
  3. Designing a robust data architecture that would persist complex workout records

This 13-week project became an exercise in constraint-based design, where each technical decision carried significant UX implications.

Technical Implementation and UX Decisions

Database Architecture

The SQLite implementation presented my first major design challenge. Early versions only stored workout names because I underestimated the complexity of relational data in mobile environments. After encountering partial data saves, I had to completely rethink my approach.

Through Logcat debugging, I identified the root cause: my initial implementation treated each data point (sets, reps, weight) as independent inserts rather than a transactional unit. This led to situations where a workout name would save but the associated exercises wouldn't persist.

The solution involved implementing proper database transactions and adding validation checks. This experience taught me that data architecture decisions directly impact user experience - what good is a workout tracker that loses your progress? I documented these learnings extensively in my Milestone 2 report, emphasizing the importance of atomic operations for data integrity.

Sensor Integration

Incorporating the accelerometer and gyroscope required careful UX consideration. While the technical implementation was straightforward (thanks to Android's SensorManager), deciding how to present this data demanded more thought.

I positioned the sensor readouts at the bottom of the workout screen after testing several layouts. This placement achieved several goals:

  • Kept the focus on workout logging (primary task)
  • Made sensor data glanceable but not distracting
  • Prevented occlusion of key interface elements

In my final report, I noted that while the sensors functioned correctly, their real value would come from better interpretation of the raw data. For example, future versions could analyze movement patterns to detect poor form or count repetitions automatically. This realization came from observing test users who glanced at the numbers but couldn't derive meaningful insights from the raw sensor output.

Maps API Challenges

The gym-finding feature evolved significantly through development. My initial approach using FindCurrentPlaceRequest failed because, as I discovered, gym locations don't have consistent tagging like restaurants or banks. This was a valuable lesson in API limitations and the importance of prototyping core features early.

After switching to SearchByText, I encountered Error 9011 - an authorization issue that persisted despite having what I believed was proper API key setup. This became an important lesson in thorough documentation review. The solution came when I realized both the legacy and new Places APIs needed enabling in Google Cloud Console, a requirement that wasn't immediately obvious from the error message alone.

This experience reinforced the importance of comprehensive API research before implementation. As noted in my final report, it also highlighted how technical hurdles can directly impact user experience - had I not resolved this, the entire maps functionality would have been unusable.

Reflections and Learnings

Developing WorkoutPal provided profound insights into the relationship between technical implementation and user experience. Several key takeaways emerged from this project:

1. Data Reliability is a UX Feature: The database issues taught me that users perceive technical reliability as part of the experience. An app that loses data feels untrustworthy, regardless of how polished the interface appears.

2. Sensor Data Needs Interpretation: While implementing the sensors was technically successful, I realized raw data means little to most users. Future implementations would benefit from processed insights rather than direct output.

3. API Limitations Shape Features: The Maps API challenges demonstrated how platform constraints can dictate feature design. This experience made me more thoughtful about vetting technical dependencies early in the design process.

As documented in my final report, these learnings have fundamentally changed how I approach mobile projects. I now prioritize data architecture planning and API research much earlier in my process, recognizing how these technical elements form the foundation of good user experiences.
